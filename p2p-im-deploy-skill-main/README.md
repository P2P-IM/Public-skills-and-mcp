# p2p-im-deploy-skill

Stable Skill layer for P2P IM deployment.

This repository contains the high-priority Agent rules for installing,
updating, repairing, resetting, and handing off P2P IM. It does not contain the
runtime installer, image tags, Docker Compose templates, plugin builds, MCP
builds, App packages, or a standalone CLI.

The mutable deployment contract and real scripts live in:

```text
https://github.com/P2P-IM/p2p-im-release
```

## Design

The deployment system has five layers:

- Skill main rules: mandatory behavior, safety stops, and completion standard.
- Skill references: stable AWS onboarding, confirmation prompts, and safety
  checklists.
- Release contract: manifest, release files, release docs, operation keys, and
  quality gates.
- Release scripts: install, update, repair, reset, status, rollback, and
  handoff executors inside `p2p-im-release/scripts`.
- Runtime resources: AWS, Docker Hub images, `p2p-as`, `p2p-mcp-server`,
  Bridge, channel plugins, App packages, and generated handoff files.

The release manifest is the machine-readable source of truth. Agents must fetch
it before any install, repair, reset, update, runtime setup, MCP setup, or App
handoff. If the public raw manifest is unavailable, authenticated GitHub access
or a verified local release checkout is allowed for controlled internal testing.
A stale local checkout is a stop condition.

## References

`SKILL.md` stays short and hard. Detailed but stable user guidance lives in:

- `references/aws-account-onboarding.md`
- `references/user-confirmation-prompts.md`
- `references/deployment-safety-checklist.md`

Provider-specific commands and verification details belong in references or the
release manifest, not in the main Skill.

## Current Policy

- Do not create a `p2p-im-cli` repository for this phase.
- Do not ask users to install a deployment CLI.
- Do not hard-code script paths in the Skill.
- Do route every operation through manifest-declared release scripts.
- Do keep release scripts structured, idempotent, testable, and future-CLI-
  compatible.

## Safety Contract

Ordinary user phrases such as "install", "reset", "upgrade", or "something is
broken" are intent signals, not execution permission. Before mutation, the
Agent must confirm target domain/node, operation type, data retention, region,
instance plan, paid actions, and release operation.

For new AWS installs, the Agent must ask whether the user already has an active
AWS account. If not, it guides account creation through the official AWS pages
and stops deployment until activation is complete. The Agent must never collect
payment details, root passwords, verification codes, MFA codes, root access
keys, AWS secret access keys, or product tokens.

The App handoff is a domain plus a current 8-character setup code or verified
QR/link generated by the release flow. Long-lived Portal password, Admin access
token, Gateway token, Agent token, legacy Channel/MCP, PEM, `.env`, database,
bootstrap credential, or setup secret values must never be handed to the user
as setup material.

No ad-hoc recovery is allowed in the standard flow. If a release script fails,
the Agent stops at that release stage and uses only release-declared recovery
actions after explicit confirmation.
